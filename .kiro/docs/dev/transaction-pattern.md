# Transaction Pattern for Multi-Write Operations

## Overview

When a service operation performs multiple database writes that must succeed or fail together, use a SeaORM transaction. This prevents orphaned records and ensures data consistency.

## Pattern

### 1. Add `_with_txn` methods to repo mutations

Each mutation that participates in a transaction needs a variant accepting `&impl ConnectionTrait`:

```rust
use sea_orm::ConnectionTrait;

impl SomeMutation {
    pub async fn create_with_txn(
        data: SomeForCreateDto,
        txn: &impl ConnectionTrait,
    ) -> Result<Uuid, DbErr> {
        let model: Model = data.into();
        let mut active_model: ActiveModel = model.into();
        active_model.not_set(Column::Id);
        let result = active_model.insert(txn).await?;
        Ok(result.id)
    }
}
```

Key steps:
- DTO ŌåÆ Model via `data.into()` (generated by `Dto` macro)
- Model ŌåÆ ActiveModel via `model.into()` (generated by `DeriveEntityModel`)
- Unset ID so DB generates it: `active_model.not_set(Column::Id)`
- Insert using the transaction reference: `active_model.insert(txn).await`

### 2. Use transaction in the service layer

```rust
use sea_orm::TransactionTrait;
use shared_shared_config::db::DB_WRITE;

let db = DB_WRITE.get().expect("DB_WRITE is not initialized");
let txn = db.begin().await.map_err(|_| AppError::Unknown)?;

// All DB writes use &txn
let id = SomeMutation::create_with_txn(dto, &txn).await?;

// Commit only after ALL writes succeed
txn.commit().await.map_err(|_| AppError::Unknown)?;

// Side effects (Kafka, HTTP calls) go AFTER commit
```

### 3. Dependency: `shared-shared-config`

The service crate needs `shared-shared-config` in `Cargo.toml` to access `DB_WRITE`:

```toml
shared-shared-config = { workspace = true }
```

## Kafka / Side Effect Ordering

**Rule:** Send Kafka messages (or any external side effects) AFTER the transaction commits.

- If Kafka fails after commit: DB is consistent. The user exists. Handle with retry or a "resend" endpoint.
- If you send Kafka before commit and the commit fails: external systems act on data that doesn't exist.

For non-critical side effects (like sending an activation email), log the failure and continue:

```rust
if let Err(e) = producer.send(&message).await {
    debug!("Kafka send failed: {:?}", e.reason);
    // Don't fail the operation - user can request resend
}
```

## Rollback

No explicit rollback is needed. If the function returns early (via `?` or `return Err(...)`) before `txn.commit()`, the transaction is automatically rolled back when `txn` is dropped.

## Example: Registration Flow

See `features/auth/service/src/authentication.rs` ŌĆö the `register` method:

1. `db.begin()` ŌĆö start transaction
2. `UserMutation::create_user_with_txn` ŌĆö create user
3. `ActiveCodeMutation::create_with_txn` ŌĆö create activation code
4. `AccessMutation::create_with_txn` ŌĆö assign role
5. `AuthCodeMutation::create_with_txn` ŌĆö create auth code
6. `txn.commit()` ŌĆö commit all writes atomically
7. `producer.send()` ŌĆö send Kafka event (non-fatal if fails)
